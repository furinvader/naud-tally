# T-014: Refine the Order Entry Screen Layout

## Status

- done

## Goal

- Polish the order entry screen so the focused room-first stepper, guest selection, and ordered-drinks-first entry flow fit cleanly on a tablet without slowing service.

## Done when

- the selected guest context stays visible while the host records orders
- the selected room and selected guest stay visible in the step header once chosen
- the current room and guest selection stay active until the host changes them explicitly from the step header
- only one workflow step body is visible at a time
- room selection, guest navigation, and order-entry controls each stay contained within the viewport
- long room lists, guest lists, and product lists scroll inside their own panels instead of forcing page scroll
- room and guest grids remain easy to scan and tap on tablet widths
- the drinks step shows only ordered drinks in first-added order instead of the full catalog
- new drink types are added from a lightweight picker instead of rendering every catalog item inline
- product controls remain fast tap targets on tablet widths, including direct numeric entry for larger quantities
- the current guest can be billed from the drinks step through a lightweight confirmation modal
- the host-tools shortcut stays visible without overtaking the main order-entry workflow
- the UX brief and task backlog record any follow-up work

## In scope

- order entry screen layout refinement
- stepper header and focused-step layout refinement
- tablet viewport behavior
- building on the shared shell and scroll primitives from [`T-023`](../done/T-023.md) instead of reintroducing route-owned chrome
- room-list or guest-selection presentation changes
- room-grid or guest-grid presentation changes
- order-entry panel layout refinement
- drinks-step clutter reduction and add-drink picker behavior
- drinks-step billing confirmation behavior
- task and UX documentation updates

## Out of scope

- route changes
- remote sync implementation
- payment processing
- mobile-specific redesign beyond preserving a usable narrow layout

## Outcome

- refined the host-operated order-entry route into a focused one-step-at-a-time tablet layout built on the shared shell and scroll primitives
- kept the selected room and guest visible in the persistent step header while leaving completed steps tappable for quick correction
- moved long room, guest, and drinks lists into owned scroll regions instead of relying on browser-page scrolling
- simplified the drinks step to show ordered drinks only, add new drink types from a lightweight picker, support direct quantity entry, and confirm billing from an inline modal
- updated the UX, design, and task docs so the refined tablet layout is recorded consistently
