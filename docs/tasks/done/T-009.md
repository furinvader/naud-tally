# T-009: Introduce Guest Tabs and Trust-Based Identification

## Status

- done

## Goal

- Replace the anonymous shared tally with a public tally view and guest-tab model that requires room number and full name.

## Done when

- the data model can represent guests with room number, full name, and drink counts
- the UI presents an active-guest public tally view with a top-bar drink and price reference
- a guest can create a tab once via the `room number -> full name` path and then continue using it without re-entering identity details
- selecting or creating a guest expands a non-modal personal tally surface on the same screen
- drink actions require a selected guest and update only that guest's counts
- explicit close and inactivity behavior intentionally reduce accidental carryover between guests
- the product, UX, and decision docs record the new model

## In scope

- guest entities
- active guest listing
- `Add yourself` room-number then full-name capture
- single-screen public tally interactions
- top-bar drink and price reference
- expandable personal tally behavior
- selected guest reset behavior
- updated state and test coverage

## Out of scope

- authentication
- payments
- live multi-device synchronization
- QR identification
- calculated billing subtotals or checkout totals
- dedicated organizer or admin surface

## Outcome

- replaced the anonymous shared tally with a single-screen public guest-tab view
- added a top-bar drink-and-price reference, an active guest list, and an inline `Add yourself` flow that follows `room number -> full name`
- introduced guest-tab persistence with room number, full name, per-drink counts, explicit close, and a 90-second inactivity reset
- extracted reusable dumb components for the price reference, stat cards, guest cards, and drink counter grid
- expanded store and component test coverage around guest creation, guest reuse, guest-specific tallying, persistence, and inactivity behavior
