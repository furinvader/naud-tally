# T-025: Separate Host-Screen UI State From Persistent Business State

## Status

- open

## Goal

- Move transient host-screen UI state out of persistent business-state owners and into the route-level host workspace.

## Done when

- selected-guest state, draft identity inputs, and comparable route-only interaction state have a clear owner in the host-workspace layer
- persistent capability state keeps ownership of only durable business data
- the state split is reflected in tests and docs

## In scope

- transient versus persistent state ownership
- route-level screen state for the host workflow
- refactors needed to stop persistent capability stores from owning route-only interaction state

## Out of scope

- changing the remote backend choice
- replacing local persistence with a new storage engine
- large UX redesigns unrelated to state ownership
